# PRD: WebGPU 流体涟漪 Demo

**版本**：v1.0  
**日期**：2026-05-21  
**状态**：待实现

---

## 1. 概览

### 1.1 项目背景

一个纯前端静态 Web 项目，利用 WebGPU Compute Shader 在手机浏览器上实时跑流体模拟。用户用手指在屏幕上滑动，产生真实感的水波涟漪效果，涟漪之间可以相互叠加干涉。

这是一个技术 Demo，用于展示 WebGPU 在移动端的计算能力，也是个人作品集的交互展示项目。

### 1.2 目标

- 在支持 WebGPU 的手机浏览器上，实现流畅的多点触控水波涟漪效果
- 保持代码简洁，不做降级兼容，只做一件事做好它
- 纯静态部署，无需后端

### 1.3 成功指标

| 指标 | 目标值 |
|------|--------|
| 移动端帧率 | 60fps（最低 30fps） |
| 首屏加载时间 | < 2s（4G 网络） |
| 触控响应延迟 | < 16ms（触点到涟漪出现） |
| 多点触控数量 | 支持 ≥ 5 个同时触点 |
| WebGPU 不可用时用户提示率 | 100%（不出现白屏/崩溃） |

## 2. 用户场景

### 场景 A：移动端主体验（核心场景）

**用户**：打开链接的手机用户（iOS Safari 17+ / Chrome Android 113+）  
**操作**：用一根或多根手指在屏幕上滑动、点击、画圆  
**期望**：每个触点实时产生水波涟漪，多指触控时涟漪叠加产生干涉纹，视觉上像真实水面被搅动

**典型流程**：
1. 打开页面，看到暗色水面背景
2. 用单指慢速划过，出现长条状涟漪尾迹
3. 双指同时点击，两个圆形涟漪扩散并相互叠加
4. 多指快速乱划，产生混沌水波效果

### 场景 B：桌面调试/展示

**用户**：开发者或桌面端访问者  
**操作**：鼠标移动（按住拖动）触发涟漪  
**期望**：鼠标拖动时产生涟漪，行为与手机触控一致，用于开发调试

### 场景 C：不兼容设备

**用户**：使用不支持 WebGPU 的旧设备/浏览器打开  
**操作**：直接访问页面  
**期望**：看到友好的提示页，告知需要什么浏览器版本，不出现白屏、报错弹窗、无响应

## 3. 功能需求

### F1：WebGPU 初始化与兼容性检测

**描述**：页面加载时检测当前环境是否支持 WebGPU，决定是进入主体验还是显示提示页。

**详细要求**：
- 检测 `navigator.gpu` 是否存在
- 调用 `navigator.gpu.requestAdapter()` 并等待结果
- adapter 存在则调用 `adapter.requestDevice()` 初始化 GPU Device
- 任何步骤失败（undefined / null / exception）均视为不支持
- 检测过程异步，期间显示加载状态（不能白屏卡住）
- 初始化成功后创建 Canvas 并配置 WebGPU Context

**技术参数**：
- Canvas 格式：`bgra8unorm`（移动端兼容性最好）
- 初始化超时：5s（超时视为不支持）

---

### F2：流体模拟核心（Navier-Stokes Compute Shader）

**描述**：GPU 上运行简化 Navier-Stokes 方程，每帧更新速度场和压力场。

**模拟参数**：
- 网格分辨率：移动端 256×256，桌面端 512×512（根据设备性能自适应）
- 时间步长：固定 1/60 秒
- 粘度系数：可调，默认 0.1
- 压力迭代次数：Jacobi 迭代 20 次/帧

**每帧 Compute Pass 顺序**：
1. **Advection**：半拉格朗日法，将速度场沿自身轨迹向后追踪，更新速度场
2. **Add Forces**：将当前帧所有输入扰动（触控/鼠标）注入速度场
3. **Pressure Solve**：Jacobi 迭代求解压力泊松方程（20 次迭代）
4. **Gradient Subtract**：从速度场减去压力梯度，保证不可压缩性
5. **Vorticity Enhancement**：计算涡旋量并加回速度场，增强旋涡细节

**双缓冲**：速度场和压力场各维护两个纹理（read/write 交替），避免读写冲突

**边界条件**：速度场边界设为零（no-slip 边界）

---

### F3：触控/鼠标输入处理（多点触控）

**描述**：将用户输入转换为速度场扰动，注入模拟核心。

**触控输入**：
- 监听 `touchstart`、`touchmove`、`touchend` 事件
- 支持同时追踪 ≥ 5 个触点（`event.touches` 全量处理）
- 每个触点记录当前位置和前一帧位置，计算速度向量
- 触点速度向量乘以力度系数后写入输入 Buffer

**鼠标输入**：
- 监听 `mousedown`、`mousemove`、`mouseup`
- 只在 mousedown 状态下的 mousemove 产生扰动
- 处理逻辑与触控一致

**坐标转换**：
- 屏幕像素坐标 → Canvas 归一化坐标 [0,1] → 网格坐标 [0, resolution]
- 处理设备像素比（devicePixelRatio）

**力度参数**：
- 触点扰动半径：网格尺寸的 3%（可调）
- 力度系数：默认 5.0（速度 × 系数注入流场）

---

### F4：视觉渲染（速度场可视化 + 水波颜色效果）

**描述**：将速度场数据渲染为可见的水波光影效果。

**渲染方式**：
- Render Pass，全屏 quad（两个三角形覆盖整个 Canvas）
- Fragment Shader 采样速度场纹理，将速度向量映射为颜色

**颜色映射**：
- 背景色：深蓝/深青色（如 `#0a1628`）
- 速度大小（magnitude）映射为亮度：静止 = 暗，快速 = 亮
- 速度方向映射为色相偏移：正 X 方向偏暖色，负 Y 方向偏冷色
- 叠加一层基于速度场梯度的高光，模拟水面反光

**视觉目标**：
- 水面感：暗色为主，涟漪区域发光
- 颜色：蓝绿色调为主，快速扰动区域出现白色高光
- 不需要光线追踪或 3D 几何，纯 2D 速度场可视化即可达到水波感

**帧率控制**：
- 使用 `requestAnimationFrame` 驱动渲染循环
- 模拟步和渲染步在同一帧内完成

---

### F5：不兼容提示页

**描述**：WebGPU 不可用时显示的替代页面。

**内容要求**：
- 标题：简洁，如「当前浏览器不支持 WebGPU」
- 说明：一句话解释这是什么（「这个 Demo 需要 WebGPU 来实时模拟流体效果」）
- 推荐浏览器列表：
  - Chrome / Edge 113+（Android / 桌面）
  - Safari（iOS 17+ / macOS 14+）
- 视觉风格：与主体验同色调，暗色背景，不显突兀
- 不显示任何 JS 报错，不显示空白 Canvas

**实现方式**：
- 默认 HTML 中只渲染提示页内容
- WebGPU 初始化成功后，JS 动态隐藏提示页、显示 Canvas

---

### F6：Nginx 部署配置

**描述**：为静态部署提供 Nginx 配置文件。

**要求**：
- 提供 `nginx.conf` 或 `nginx-site.conf` 配置片段
- 启用 Gzip 压缩（针对 JS/WGSL 文件）
- 设置正确的 MIME type（`.wgsl` 文件 → `text/plain`）
- 缓存策略：HTML 不缓存，JS/CSS/WGSL 长缓存（1 年 + contenthash）
- HTTPS 兼容（WebGPU 要求 Secure Context）

## 4. 非功能需求

### 4.1 性能

| 设备类型 | 目标帧率 | 最低帧率 | 网格分辨率 |
|----------|----------|----------|------------|
| 高端手机（iPhone 15 / Pixel 8） | 60fps | 45fps | 512×512 |
| 中端手机（iPhone 12 / 一般 Android） | 60fps | 30fps | 256×256 |
| 桌面（Chrome / Edge） | 60fps | 60fps | 512×512 |

- 单帧 GPU 计算时间目标：< 8ms（移动端）
- 内存使用：< 200MB（不触发移动端低内存警告）

### 4.2 兼容性

**支持**：
- iOS Safari 17+（iPhone 12 以上型号）
- Chrome for Android 113+
- Chrome 桌面 113+
- Edge 桌面 113+

**不支持（显示 F5 提示页）**：
- Firefox（截至 2026-05，WebGPU 仍实验性）
- iOS 16 及以下
- 任何不支持 WebGPU 的设备

### 4.3 包体积

- 总 JS 体积（gzip 后）：< 50KB（无框架依赖，纯 WebGPU 原生）
- WGSL Shader 文件：内联或单独文件均可，不影响包体积

### 4.4 可维护性

- WGSL Shader 与 JS 逻辑分离，shader 代码独立文件
- 模拟参数集中在一个配置对象，方便调参
- 无外部框架依赖（无 Three.js / Babylon.js），纯原生 WebGPU

## 6. 排除范围

以下内容明确**不做**：

| 排除项 | 原因 |
|--------|------|
| WebGL2 降级回退 | 增加代码复杂度，保持简洁 |
| 3D 几何/光线追踪水面 | 超出 Demo 范围，2D 可视化已足够 |
| 音效（水声） | 与技术目标无关 |
| 截图/录制功能 | 非核心体验 |
| 参数调节 UI（滑块等） | 参数在代码内调，不需要用户界面 |
| 多场景/主题切换 | 单一水波效果即为完整体验 |
| PWA/离线缓存 | 过度工程 |
| 性能分析面板 | 开发时用 Chrome DevTools 即可 |
| 后端/API | 纯静态项目 |
| 用户数据存储 | 无需登录/存档 |

## 7. 风险与缓解

### 风险 1：移动端 WebGPU 帧率不达标

**可能性**：中  
**影响**：主体验卡顿，失去水波感  
**缓解方案**：
- 网格分辨率根据设备自适应降级（256→128）
- Jacobi 迭代次数在低端设备减少（20→10）
- 通过 `device.limits` 检测设备能力动态调参

### 风险 2：iOS Safari WebGPU 兼容性问题

**可能性**：中（Safari WebGPU 实现较新，edge cases 多）  
**影响**：iOS 用户无法体验  
**缓解方案**：
- 实机测试 iPhone 12 / 14 / 15
- 避免使用 Safari 不支持的 WebGPU 高级特性
- 如遇 Safari 特有 bug，优先 workaround 而非放弃

### 风险 3：WGSL Shader 调试困难

**可能性**：高（GPU shader 无断点调试）  
**影响**：开发效率低，数值问题难定位  
**缓解方案**：
- 每个 Compute Pass 单独实现和测试
- 先跑单步模拟验证物理正确性，再组合完整 pipeline
- 使用 `debugValueBuffer` 技术将 GPU 中间值读回 CPU 检查

### 风险 4：流体模拟数值不稳定（发散）

**可能性**：中  
**影响**：速度场变 NaN/Inf，画面崩溃  
**缓解方案**：
- 半拉格朗日 advection 自带数值耗散，天然稳定
- 每帧对速度场做 clamp（限制最大速度）
- 时间步长固定（不跟随帧率变化），避免大步长不稳定

### 风险 5：Canvas 尺寸/DPR 适配问题

**可能性**：低  
**影响**：在高 DPR 设备上模糊或性能过高  
**缓解方案**：
- Canvas 内部分辨率与模拟网格解耦（网格固定，Canvas 独立适配屏幕）
- devicePixelRatio 最高取 2（超过 2 的设备也只用 2）
